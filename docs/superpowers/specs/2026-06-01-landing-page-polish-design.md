## Manifest inlining strategy

**Problem:** `fetch('audio/book-5/manifest.json')` fails on `file://` (CORS-equivalent restriction). Page shows "Audio manifest not found yet" — even though the file is right next to it.

**Solution:** Embed the manifest as a JSON `<script type="application/json" id="manifest-fallback">…</script>` block in `index.html`. Modify `init()` to:

```
1. Try fetch(M_URL, {cache:"no-store"})
2. On failure (any throw, including TypeError on file://), fall back to JSON.parse(document.getElementById('manifest-fallback').textContent)
3. If both fail, show the existing empty state.
```

Trade-off: page size grows by ~10–15 KB (90 chapter records). Acceptable — the file is already 27 KB and we add ~50% on top, all gzippable. Manifest is regenerated by `build_manifest.py`; we need a small shell step to refresh the inlined copy on each manifest rebuild. Encode that step as a one-line shell snippet in `audiobook/README.md` so future renders update both copies.

The fetch-first order means: when hosted, the live `manifest.json` still wins (so updates to chapter readiness without a re-build of `index.html` still work). The inlined copy is a fallback for `file://` and a first-paint speedup.
